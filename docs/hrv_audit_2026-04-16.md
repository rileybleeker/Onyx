# HRV Analysis Pipeline — End-to-End Audit

**Date:** 2026-04-16
**Scope:** Read-only audit of the HRV analysis pipeline (`hrv_analysis.py`, `hrv_predict.py`, supporting ETL, GitHub Actions, Supabase `pds` schema).
**No code, SQL, or workflow changes made in this audit.** Ad-hoc verification scripts live in `analysis_output/audit_scratch/` and are gitignored.

---

## Executive Summary

**Is the descriptive analysis (Phases 1–6) trustworthy as-is?** **No.**

The underlying HRV target is reliable (WHOOP RMSSD, N=534, no obvious distribution pathology), but the correlation rankings and journal-impact conclusions that the pipeline produces are quantitatively unreliable. Three Critical issues stack:

1. Stage 1 Spearman screening runs with no BH-FDR — with ~200 features tested, ~10 of the "top drivers" are expected to be false positives at α=0.05.
2. Stage 3 "standardized OLS with f²" described in docs is **not implemented** — the code runs Welch t-tests on journal flags and stops there. Effect sizes are not adjusted for between-behavior correlation.
3. Garmin sleep timestamps (`sleep_start` / `sleep_end`) are stored as local-clock values mislabeled as UTC (`garmin_etl.py:234`). Any bedtime / sleep-midpoint / time-of-day feature is off by ~4-5 hours.

Add to that one Critical item: 15 canonical Garmin training-state predictors (ACWR, training load, HRV factor, training readiness) are silently dropped as sparse (Phase 6.2). *(The original audit also flagged WHOOP journal `cycle_date` semantics as Critical-if-verified; verification on 2026-04-16 refuted this — see Phase 4.5 update.)*

**Are the predictive models (Phase 7) production-ready?** **No.**

- XGBoost beats the best baseline (`baseline_7d_avg`) by 2.62 ms MAE (~9%) on 333 walk-forward predictions. Usable as-is *only* for absolute-value estimates. For directional guidance ("will HRV be up or down tomorrow?") the code's stored directional-accuracy metric is 53.3%, barely above chance. A cleaner definition pulled from the stored predictions gives 61-82% monthly, but the pipeline reports the weaker number on the dashboard.
- **Prophet MAE = 42.11 ms is worse than persistence (32.76 ms).** Prophet is a net negative in the ensemble at current tuning. Remove or re-tune.
- **XGBoost's 90% prediction intervals achieve 24.7% actual live coverage** (stored `ci_coverage` for eval_date 2026-04-16). The interval width is fit on the test set residuals and treated as a fixed σ in live prediction — a tautology at training time, badly miscalibrated in production.
- Drift detection is labelled "30-day rolling MAE" but actually aggregates every prediction since May 2025. The metric desensitizes as history grows and won't trigger on recent-only degradation.
- No Diebold-Mariano test, no residual Ljung-Box, no permutation-importance cross-check, no controllable-only SHAP ranking, no prediction-stability diff on retrain.

Verdict: the current XGBoost model produces correct-order-of-magnitude next-day HRV estimates and is ~9% better than the best trivial baseline — good enough for a dashboard visual, **not** good enough to drive recovery recommendations. Tighten calibration + leakage + drift-metric issues before adding downstream actions.

### 1.4 Modeling N and dimensionality

- **Target N** (non-null `whoop_hrv_rmssd`): **534 days**, 2024-10-28 → 2026-04-16.
- **Distribution**: mean = 126.52 ms, SD = 33.18 ms, range [36.25, 214.67]. Positive, wide range — consistent with RMSSD in ms.
- **Feature matrix size**: 268 engineered features claimed (per `project_onyx_hrv.md`). Verify in Phase 6.
- **Observations-per-predictor ratio**: 534 / 268 ≈ **2.0**. Rule of thumb for linear regression is 10–20 obs/predictor — this dataset is **5–10× under-powered** for OLS/partial correlation without dimensionality reduction. **[Critical]** for Stage 2/3 analytical stability; XGBoost is tolerant because of L1/L2 regularization but SHAP attributions at N=534 across 268 features are still noisy.
- Recommendation: before Stage 2, apply a screening filter (e.g., keep only Stage-1 BH-FDR survivors, typically 20–50 features) OR apply PCA / elastic-net for selection.

## Phase 2 — Analytical Correctness

### 2.1 Stage 1 — Spearman screening

`hrv_analysis.py:923-941` computes Spearman correlations between every numeric feature and `hrv_next` (tomorrow's HRV via `.shift(-1)` at line 913). Results are sorted by `abs(r)`.

**Normality & Spearman defensibility** (scratch `analysis_output/audit_scratch/phase2_stats.py`, N=534):

| Test | Statistic | p | Interpretation |
|---|---:|---:|---|
| Shapiro-Wilk on raw HRV | W=0.994 | 0.027 | Marginally non-normal; skew=0.11, kurt=−0.51 — near-Gaussian |
| Shapiro-Wilk on log(HRV) | W=0.976 | 1.2e-7 | log-transform makes it WORSE — do **not** log HRV |

Spearman is acceptable as a rank-based screening tool. **No action needed** on the choice of Spearman vs. Pearson — but the subsequent Pearson heatmap (`hrv_analysis.py:951`) should be interpreted knowing the raw distribution is only approximately Gaussian.

**BH-FDR missing — [Critical].** Grep for `fdrcorrection`, `multipletests`, `benjamin` returns zero matches in `hrv_analysis.py`. The audit prompt calls this a "Spearman + BH-FDR" stage; the code only sorts by abs(r) and stores raw p-values. With ~200+ features tested, expected false discoveries at α=0.05 ≈ 10, many of which will appear in the "top drivers" table. Before downstream stages, BH-FDR (q≤0.05) should be applied and only surviving features passed to Stage 2.

**α / q-threshold documentation**: not present anywhere in the file. The `head(25)` / `head(15)` / `head(10)` cutoffs at lines 968, 994, 1092 are magic numbers, not a statistical threshold.

### 2.2 Stage 2 — Partial correlation

`hrv_analysis.py:988-1015`:

```python
controls = [c for c in ["hrv_lag1", "whoop_sleep_duration_milli"]
            if c in hrv_valid.columns]
top15_feats = corr_df.head(15)["feature"].tolist()
for feat in top15_feats:
    ...
    r_partial = pg.partial_corr(data=sub, x=feat, y=STAT_TARGET, covar=controls)
```

- **Controls for only 2 variables** (`hrv_lag1`, sleep duration), not "all Stage 1 survivors simultaneously" as the audit design calls for. With ~20+ plausible confounders (sleep stages, strain, alcohol, caffeine), the remaining confounding is substantial. **[Critical]**
- **Applied only to the top 15 by Stage 1 abs(r)** (line 994) — because Stage 1 has no FDR, the top 15 may include false-positive Spearman hits.
- **No VIF check** before partial correlation. If controls are collinear with the target feature, partial r is unstable.
- **No FDR at Stage 2 either** — each of the 15 partial correlations returns a p-value, none corrected.

**ACF computed on full dataset** — line 1076: `hrv_series = hrv_valid[STAT_TARGET].dropna().values`. `hrv_valid` is the full (train + test) data. The ACF is then used to choose the SARIMAX lag structure and informs the 60-day rolling-correlation window (line 1126). Measured impact of this look-ahead:

| Lag | ACF on full series | ACF on training-only (70%) | Δ |
|---|---:|---:|---:|
| 1 | 0.272 | 0.271 | 0.001 |
| 7 | 0.136 | 0.080 | **0.056** |

The lag-1 autocorrelation is stable; the lag-7 (weekly-cycle) ACF is **70% stronger on the full series than on training data alone**. Any feature-engineering or SARIMAX order choice that leans on lag-7 structure is modestly biased by look-ahead. **[Medium]**

### 2.3 Stage 3 — Standardized OLS with f² — **does not exist in the code**

`hrv_analysis.py:1017-1145` is **not** a standardized-OLS regression. The block is:

1. **Journal conditional analysis** (lines 1017-1071): Welch's unequal-variance t-test between "yes"/"no" groups for each boolean journal feature, plus Cohen's d and a 95% Wald CI on the mean difference. No OLS, no standardization, no f². Multiple comparisons across ~40 journal features are uncorrected.
2. **ACF/PACF plots** (lines 1074-1087).
3. **Granger causality** (lines 1089-1113) for top-10 features at maxlag=3, p-values not corrected for the 30 tests.
4. **Rolling correlation plot** (lines 1115-1142).

Grep for `StandardScaler`, `\bols\b`, `f_squared`, `f2_` → **zero matches in `hrv_analysis.py`**. The "three-stage pipeline with standardized β and f²" described in project docs/prior memory is **not implemented**. **[Critical]** — either document what IS implemented (one-stage screening + t-tests + Granger) or implement Stage 3.

Consequences:
- Reported "drivers" are ranked by raw Spearman r, not by adjusted effect size.
- Journal effect sizes are uncorrected for between-behavior correlation (alcohol + late meal co-occur; their independent contributions are not separated).

### 2.4 Residual / autocorrelation diagnostics — missing

Grep for `durbin_watson`, `shapiro`, `ljungbox`, `acorr_ljungbox` in `hrv_analysis.py` → **zero matches**. No residual diagnostics run on any model's fit.

Scratch run on the persistence baseline (ŷ(t)=y(t−1)):

| Test | Value | Interpretation |
|---|---:|---|
| Durbin-Watson on Δy | 2.91 | Negative autocorrelation in deltas → mean-reversion / overshoot. DW > 2.5 violates OLS iid assumption; use HAC SEs if OLS is added. |

### 2.5 Leakage scan

Grep/inspect of `prepare_ml_data` (`hrv_analysis.py:1152-1180`):

```python
exclude = {"calendar_date", "hrv_target_t1", "hrv_next",
           "whoop_recovery_score"}  # derived from same sleep's HRV — circular
```

- **`whoop_recovery_score` correctly excluded** — it is computed from the same night's HRV and would circular-leak the target.
- **`whoop_hrv_rmssd` (TARGET) is NOT excluded** — kept as a feature (comment at lines 1162-1164 explains this is "this morning's HRV score, known at prediction time, = `hrv_lag0`"). Since target = `HRV(t+1)` via shift(-1), using `HRV(t)` as a feature is equivalent to `hrv_lag1` in the target's frame and is not leakage.
- **`garmin_hrv_*` columns**: these are from last night's sleep (measured ~7-8h before "tomorrow's" HRV prediction target). Also valid.
- **WHOOP sleep metrics** (lines 1162-1170 comment): "represent yesterday's sleep quality, valid context for tonight's prediction." But the lag semantics here are subtle — see Phase 4.5 for date-alignment verification.

**Potential leakage not flagged**:
- `hrv_analysis.py:1319-1321` — prediction intervals are computed as `test_pred ± 1.645 * std(residuals_on_test_set)`. That is, the CI width is **fit on the test set** and the coverage is then measured on the same test set. The claimed "90% CI coverage" in `test_metrics` is therefore tautological by construction (residual std on a zero-mean set yields ~68% within ±1σ, ~90% within ±1.645σ). This metric carries no real calibration signal — for live `hrv_predict.py` predictions it's used as a fixed radius, whose live coverage has never been measured. **[High]**

### 4.3 Timezone handling

- **Storage layer**: every `*_at` and `sleep_start`/`sleep_end` column is `timestamptz`. Postgres stores these as UTC instants; display TZ is session-dependent.
- **Garmin `sleep_start`/`sleep_end` — stored as local-clock-labeled-UTC — [Critical]**:

  `garmin_etl.py:234-235`:
  ```python
  "sleep_start": datetime.fromtimestamp(sleep_start / 1000, tz=timezone.utc).isoformat()
  ```
  Garmin's API returns `sleepStartTimestampLocal` (the field pulled at `garmin_etl.py:227`): this is **epoch-milliseconds offset to the user's local clock**, not UTC. Parsing it with `tz=timezone.utc` produces a `datetime` that prints like `Apr 16 01:51 UTC` but actually represents `Apr 15 21:51 ET`. The real UTC of the event is `Apr 16 05:51 UTC` — a **4–5 hour offset**.

  Verified empirically:

  | `calendar_date` | Stored `sleep_start` | "UTC hour" | "ET hour" |
  |---|---|---:|---:|
  | 2026-04-16 | `2026-04-16 01:51:24+00` | 1 | 21 (Apr 15) |
  | 2026-04-15 | `2026-04-15 00:43:00+00` | 0 | 20 (Apr 14) |
  | 2026-03-25 | `2026-03-25 02:00:24+00` | 2 | 22 (Mar 24) |

  Computed "ET hour" of 20–22 = 8–10 PM would mean Riley is falling asleep before his usual bedtime, which is implausible. The stored value is the local clock reading treated as UTC. Any feature engineering that reads `sleep_start` as a UTC instant — e.g., `bedtime_hour = EXTRACT(HOUR FROM sleep_start AT TIME ZONE 'America/New_York')` — will produce a value **4h too early** in EDT (5h in EST). **[Critical]** — this silently corrupts every sleep-timing feature in `build_feature_matrix()` that touches Garmin sleep timestamps.

- **WHOOP `start_time`/`end_time`**: Stored as genuine UTC (WHOOP API returns ISO-8601 with `Z`). The view correctly casts to UTC date. No bug.
- **Eight Sleep**: ETL uses `EIGHTSLEEP_TIMEZONE = America/New_York` (`eight_sleep_etl.py:36,62,108`) when deriving `calendar_date` from session-end. Correct.

### 5.6 Analysis reproducibility — [Critical]

`hrv_analysis_results` schema:

| Column | Type |
|---|---|
| id | integer |
| result_type | text |
| result_key | text |
| result_json | jsonb |
| computed_at | timestamptz |

**No `input_data_hash`, no `feature_set_hash`, no `training_row_count`, no `source_snapshot_ids`**. Grep for `hashlib`, `sha256`, `md5`, `input_data_hash` across the repo → only `whoop_etl.py` imports `hashlib` (for OAuth PKCE, not data). `hrv_analysis.py` computes no hash.

`hrv_model_metrics` stores `model_version` (= `f"{date.today().isoformat()}_v1"` at `hrv_analysis.py:111`) but nothing tying the version to the exact input data. `hrv_predictions` stores `training_window_start/end` and `model_version`, which is a partial signal but doesn't detect revised source data (e.g., a backfilled WHOOP cycle from 3 weeks ago).

**Recommendation**:

1. Add `input_data_hash TEXT` to `hrv_analysis_results`, `hrv_model_metrics`, `hrv_predictions`.
2. Compute the hash in `hrv_analysis.py` from the sorted CSV dump of `X, y` (after `prepare_ml_data`) using `hashlib.sha256`.
3. Store alongside each row. The GitHub Actions retrain job logs the hash to the workflow summary so two runs can be diffed.

### 7.7 Drift detection (`hrv_predict.py:312-335`)

```python
DRIFT_THRESHOLD = 1.5
if current_mae > DRIFT_THRESHOLD * baseline_mae:
    log.warning(f"MODEL DRIFT DETECTED: current 30d MAE={current_mae:.1f}ms, ...")
```

Issues:

- **"30-day rolling metric" is computed across ALL history**, not 30 days. `recompute_rolling_metrics` (lines 262-309) reads every row from `hrv_predictions`, drops NaN, and computes MAE on the entire dataset for each model. There is no date filter. The function name is misleading and the metric loses responsiveness as history grows. **[High]**
- **Fixed threshold `1.5 × backtest_mae`** → fires on naturally harder months. 2026-03 monthly MAE = 32.08 ms; backtest_initial MAE ~ 24 ms. Ratio 1.34 — below threshold. But a month 15% worse than backtest could shift to >1.5 and trigger false retrain. Suggested replacement: retrain if rolling 14-day MAE > P95 of historical monthly MAE for 3 consecutive days.
- **Baseline lookup** (line 317-319) finds `model_version='backtest_initial'` for XGBoost. Only 2 rows exist (eval_date 2026-04-04 and 2026-04-05). Code uses `baseline_rows.data[0]` — an arbitrary one of the two. Should be explicit (e.g., min eval_date, or stored stats on the initial backtest). **[Medium]**
- Drift-alert row inserted into `pds.sync_log` (line 329-335). Downstream dashboard doesn't subscribe. Alerts are logged but silent to the user. **[Medium]**
